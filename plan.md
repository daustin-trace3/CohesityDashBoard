# Design: Cohesity Backup Dashboard — Wow Layer

## Problem Statement

An existing Cohesity Backup dashboard with tables, charts, and animations that feels
flat for customer demos. Mixed audience (backup admins, IT managers, executives).
The goal: add a "wow" layer that lands in under 5 seconds for any audience, without
narration — and tells a predictive story the existing Cohesity UI doesn't tell.

## What Makes This Cool

Three features that compose into one visual story:

> "If ransomware hit right now, how screwed are we?"

No other Cohesity dashboard answers that question. This one does — at a glance.

## Constraints

- Data comes from Cohesity API, polled (not pure real-time; assumed 5-minute cycle)
- Mixed demo audience: exec needs instant comprehension, admin needs actionability
- Prediction logic must be visually legible — no black-box scores
- Demo must not panic the customer — red squares need context framing
- Scale: if >50 workloads, Recovery Exposure Map groups by type with collapsible sections

## Premises

1. The dashboard feels flat because there's no anchor visual — tables and charts of
   equal weight give the eye nowhere to land.
2. For a mixed demo audience, the wow has to land in under 5 seconds without
   explanation. If you narrate it, you've lost the executive.
3. "Predictive" for backup is most powerful framed as recovery risk, not job health —
   customers fear what they'd LOSE, not whether a job ran slow.
4. Prediction logic needs to be visually legible — a score with visible inputs beats
   a black-box AI recommendation.

## Approaches Considered

### Approach A: Protection Score
Single dominant gauge/ring showing 0-100 score computed from job success rate, RPO
gap, storage headroom, and failure trends. Everything else becomes evidence for or
against this number. Effort: S (1-2 days). No new data needed.

### Approach B: Recovery Exposure Map
Heatmap grid of protected workloads. Each cell = a VM/server/app, colored by
recovery exposure (time since last clean recovery point × criticality). Red = exposed,
green = protected. Hover shows: last backup, RPO gap, estimated time-to-recover.
Effort: M (3-5 days). Needs RPO gap model.

### Approach C: Predictive Job Risk Timeline
Sparkline next to each backup job showing duration trend over last 7-14 runs. Jobs
trending longer get flagged "at risk of missing window." "N jobs at risk tonight"
as a prominent widget. No ML — linear trend detection. Effort: M (3-4 days).
Needs historical duration storage.

## Recommended Approach

**All three, composed into a single layout:**

```
┌─────────────────────────────────────────────────────────────────┐
│  PROTECTION SCORE: 74/100   [RPO] [Jobs] [Storage] [Compliance] │  ← A
├─────────────────────────────────────┬───────────────────────────┤
│                                     │  ⚠ 3 JOBS AT RISK TONIGHT │  ← C
│   RECOVERY EXPOSURE MAP             │                           │
│                                     │  Job A  ▁▂▃▄▅▆  ⚠ +18%  │
│   [VM][VM][VM][VM][VM]              │  Job B  ▂▂▂▂▂▂  ✓ stable │
│   [DB][DB][DB][DB][DB]              │  Job C  ▁▁▂▃▄▅  ⚠ +31%  │
│   [NAS][  ][NAS][NAS][  ]           │  Job D  ▃▃▃▃▃▃  ✓ stable │
│                                     │                           │
│   Red = exposed  Green = protected  │  [view all jobs →]        │
├─────────────────────────────────────┴───────────────────────────┤
│  [existing tables / job history / storage charts]               │
└─────────────────────────────────────────────────────────────────┘
```

**Visual flow by audience:**

- **Executive:** Sees 74/100 Protection Score → sees red squares in the map →
  "what are those?" → clicks → "oh, those are my most exposed workloads." Done.
  No narration. Total time to comprehension: ~10 seconds.

- **Admin:** Sees "3 jobs at risk tonight" → scans sparklines → knows what to
  fix before the backup window. Actionable before the exec even asks a question.

- **Manager:** Sees score dropping → checks sub-scores (RPO compliance is the
  gap) → has a number to put in the next status report.

**Build order (sequential, not parallel):**
1. **Week 1:** Protection Score (A) + data persistence spike (confirm backup window
   source + start storing job duration history — both are prerequisites for Week 2)
2. **Week 2:** "Jobs at Risk" widget + sparklines (C) — only after Week 1 spike resolves
3. **Week 3+:** Recovery Exposure Map (B) — the hero visual that makes it different

**Demote existing tables below the fold.** The three new components occupy the full
viewport on initial load; users scroll for detailed job/history tables.

## Implementation Notes

### Protection Score (A)
- Inputs: job success rate (last 7 days), worst RPO gap across all workloads,
  storage headroom (% to threshold), compliance pass rate
- Formula (committed): weighted average — RPO gap 40%, job success rate 30%,
  storage headroom 20%, compliance 10%. Weights live in a named constants object
  (`SCORE_WEIGHTS`) — single source of truth, easy to tune without hunting.
  RPO weighted highest because it directly measures recovery exposure, the central
  risk the dashboard communicates.
- If compliance data unavailable from API: redistribute weight proportionally
  (RPO 47%, jobs 35%, storage 24%, compliance 0%) rather than silently returning
  0 for the compliance component. Document the redistribution in a code comment.
- Show sub-scores on hover/click so the number is auditable, not a black box
- On initial page load, animate the ring filling to its current value — this makes
  the score feel alive and draws the eye without requiring narration
- "Last updated X min ago" staleness indicator near the score header. Turns yellow
  if data is >5 min old, red if >10 min old. On API error: stale-on-error (keep
  last known score + red badge showing "API unreachable").
- Visual: large ring gauge, color transitions green→yellow→red at 80 and 60

### Recovery Exposure Map (B)
- X-axis: workload type grouping (VMs, DBs, NAS)
- Cell color: hours since last clean recovery point (green <4h, yellow 4-24h,
  red >24h, GRAY = no backup data / null timestamp). Four states total.
  Criticality is NOT used for color; it affects cell SIZE only, so the map works
  without criticality data.
- Gray cell: workload has never had a successful backup, or API returns null
  for last recovery point. Tooltip: "No backup data — this workload may be
  unprotected." Prompts useful conversation in demos.
- If criticality unavailable from API (resolve in Week 1 spike): all cells render
  at equal size; map still works, just loses tier differentiation
- Hover tooltip: "Last backup: 6h ago | RPO gap: 2h | Est. recovery time: 45 min"
- Label framing: "Recovery Exposure" not "Backup Status" — exposure is the risk frame
- Framing context: legend reads "Red = >24h since clean recovery point — not a
  failure, a risk to review." Appears on initial load for 3 seconds, then on legend
  hover. Prevents customer panic.
- Scale: if >50 workloads, group by type (VMs/DBs/NAS) with collapsible sections.
  Load first 20 per type on initial render with "show all" expand.

### Predictive Job Risk Timeline (C)
- **Hard prerequisite:** backup window per job MUST come from the Cohesity API.
  If the API does not expose a backup window for a job, the at-risk flag is DISABLED
  for that job. No default 24h assumption — a false positive is worse than a missing
  flag. The "Jobs at Risk Tonight" widget only counts jobs with API-backed windows.
- Store job duration on each poll cycle. Persistence: SQLite backend (confirmed —
  dashboard has a backend process). Do NOT rely on in-memory state (wiped on restart).
  Window: retain last 50 runs or 30 days per job, whichever is less.
- Trend detection: shared `computeTrend(durations[])` utility function. Returns
  `{slope, isAtRisk, regressionLine[]}`. Both the sparkline visualization and the
  at-risk flag consume this function — they cannot drift.
  Algorithm: linear regression over most recent 10 completed runs, normalized by
  wall-clock time (slope in min/day, not min/run). Flag if trend exceeds +10%/day
  of baseline median.
- "At risk" definition: job trending to exceed its backup window by >15% within
  the next scheduled run — computed only when backup window is API-confirmed.
- Sparkline shows raw duration per run with regression line on hover — the trend
  is self-evident without explanation.
- "N jobs at risk tonight" widget: prominent, yellow/red count badge. Shows "N/A"
  for jobs where backup window is not available from API.

## Week 1 Spikes (resolve before Week 2 starts)

These are blockers — Week 2 cannot ship without answers to all three:

1. **Criticality tiers:** Does Cohesity API expose criticality/tier tags per workload?
   If not: default to equal cell sizes in the Exposure Map (still works). Record the
   answer and update Implementation Notes accordingly.

2. **Job duration history:** Is it stored across poll cycles today? If not: add
   persistence in Week 1 (local DB, SQLite, or file). Choose a window: store 30 days
   or 50 runs per job, whichever is less.

3. **Backup window per job:** Does Cohesity API return the job's scheduled window?
   If not: default assumption is 24h window. Surface this assumption in the "at risk"
   tooltip ("assumes 24h backup window — update in settings").

4. **Live vs. demo data:** If using synthetic data for demos, define 2-3 demo
   scenarios with concrete values seeded at startup:
   - Scenario A (healthy): score 87, 1 at-risk job, 2 workloads yellow on map
   - Scenario B (warning): score 62, 4 at-risk jobs, 6 workloads red on map
   - Scenario C (critical): score 41, 9 at-risk jobs, many red — dramatic for demos

## Success Criteria

- An executive who has never seen the dashboard before says what the Protection Score
  represents within 10 seconds, unprompted (observable: they describe it before you
  explain it)
- The Recovery Exposure Map shows at least one workload with >24h since last clean
  recovery point in the demo dataset; the exec asks about it without being prompted
- "Jobs at risk tonight" count is non-zero in the demo; at least one sparkline shows
  a visible upward trend (not just a flat line with a flag)
